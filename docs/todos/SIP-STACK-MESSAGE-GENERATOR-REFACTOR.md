# SIP Stack Message-Generator Refactor — Plan

**Status:** Not started
**Driver:** Eliminate ad-hoc SIP message construction from B2BUA / transaction layer. All message construction moves behind a small set of "correct by default" pure generators in the SIP stack. B2BUA code stops knowing how ACK / CANCEL / PRACK / in-dialog requests are assembled.
**Owner:** TBD
**Rough cost:** ~2 weeks focused effort, ~12 PRs (one per slice).

Only edit this document if a new decision contradicts what is written here.

---

## Why this exists

Today SIP message construction is scattered across three layers:

- [`src/sip/MessageFactory.ts`](../../src/sip/MessageFactory.ts) exposes ~15 builder functions (`buildBLegInvite`, `buildAck`, `buildRelayedAck`, `buildBye`, `buildRelayedBye`, `buildCancel`, `build200Ok`, `build487`, `build491`, `buildRejectResponse`, `build100Trying`, `buildOptions`, `buildPrack`, `buildRelayedPrack`, `buildNotify`, `buildOriginatedInvite`, `buildRelayedRequest`). Each emits `Via` and `Contact` as the literal string `"__PLACEHOLDER__"`.
- [`src/sip/SipRouter.ts`](../../src/sip/SipRouter.ts) (`stampHeaders`) rewrites placeholders to the correct Via/Contact using local address + custom `cr`/`lg`/`em`/`callRef`/`leg`/`emerg` params. Also generates the Via `branch` and captures it back into `leg.inviteBranch` / `dialog.ackBranch` for later CANCEL / ACK-retransmit reuse.
- [`src/sip/TransactionLayer.ts`](../../src/sip/TransactionLayer.ts) synthesises 100 Trying, auto-200 to CANCEL, auto-487, and auto-ACK for non-2xx. Auto-ACK is its own private `buildAckForNon2xx` — duplicate of ACK construction. UAS To-tag pinning (§12.1.1) lives here as `txn.uasToTag`.
- [`src/b2bua/rules/framework/ActionExecutor.ts`](../../src/b2bua/rules/framework/ActionExecutor.ts) is the heaviest consumer (~15 build sites) and owns the per-method CSeq math: `dialog.localCSeq + 1` for BYE/re-INVITE/PRACK/NOTIFY/…, `dialog.lastInviteCSeq` for ACK-for-2xx.

Problems this creates:

1. **`"__PLACEHOLDER__"` sentinel** — intermediate `SipRequest` / `SipResponse` objects are syntactically valid but semantically broken. "Correct by default" is violated.
2. **Duplicate ACK construction** — `buildAckForNon2xx` in `TransactionLayer` re-implements ACK independently.
3. **Fifteen builders with overlapping shapes** — "fresh" vs "relayed" vs "response-from-request" all live side-by-side. Hard to audit against RFC 3261.
4. **CSeq knowledge scattered** — ACK-for-2xx reads `dialog.lastInviteCSeq`, a field that's easy to forget to update and duplicates state already on the INVITE client transaction. PRACK-between-INVITE-and-2xx is the concrete foot-gun (`dialog.localCSeq` has moved on, so `lastInviteCSeq` is the only way).
5. **Branch ownership ambiguous** — `stampHeaders` generates it, `leg.inviteBranch` / `dialog.ackBranch` cache it, CANCEL-reuse and ACK-retransmit-reuse are bespoke call sites.

**Target invariant after this refactor:**

1. **Zero SIP message construction lives outside `src/sip/`.** Every outbound request / response in the B2BUA is produced by a stack generator.
2. **Generators are pure functions.** All "random mandatory" parameters (Call-ID, branch, tags, local address, CSeq number when applicable) are arguments. No Effect dependencies.
3. **Generators are correct-by-default.** The returned message is immediately sendable; no placeholder strings, no post-processing step.
4. **Single source of truth per identifier.** Branch generated by `newBranch()` at the call site that will persist it. `CSeq` managed by the stack (returned in updated dialog). UAS To-tag pinned by `TransactionLayer` for internally-synthesised responses; caller-provided for application-level responses. INVITE CSeq source is the `InviteClientTransactionHandle`, not a duplicated `lastInviteCSeq` field.
5. **The SIP-level Dialog state lives in the stack.** `src/sip/Dialog.ts` holds `StackDialog`. The B2BUA's `Dialog` is `{ sip: StackDialog, ext: B2buaDialogExt }` — composition forces every existing `dialog.localCSeq` / `dialog.toTag` / `dialog.callId` call site to be migrated (compile error until fixed).

## Target API

### Types (stack-public)

```ts
// src/sip/Dialog.ts — RFC 3261 §12 dialog state, stack-owned
export interface StackDialog {
  readonly callId: string
  readonly localTag: string          // our tag on this leg
  readonly remoteTag: string         // peer tag on this leg
  readonly localUri: string          // From-URI when we send in this dialog
  readonly remoteUri: string         // To-URI
  readonly remoteTarget: string      // peer Contact URI — Request-URI for in-dialog
  readonly localCSeq: number         // last-sent sequence number
  readonly routeSet: ReadonlyArray<string>  // from Record-Route (§12.1.2)
}

// src/sip/TransactionLayer.ts — exposed transaction handles
export type ClientTransactionHandle =
  | InviteClientTransactionHandle
  | NonInviteClientTransactionHandle

export interface InviteClientTransactionHandle {
  readonly kind: "invite"
  readonly branch: string
  readonly originalInvite: SipRequest
  readonly destination: { readonly host: string; readonly port: number }
}

export interface NonInviteClientTransactionHandle {
  readonly kind: "non-invite"
  readonly branch: string
  readonly originalRequest: SipRequest
  readonly destination: { readonly host: string; readonly port: number }
}

// src/sip/generators.ts — structured header input shapes
export interface ViaSpec {
  readonly localIp: string
  readonly localPort: number
  readonly transport: "UDP" | "TCP" | "TLS" | "WS" | "WSS"
  readonly branch: string
  readonly customParams?: Record<string, string>  // { cr, lg, em } — opaque to stack
}

export interface ContactSpec {
  readonly user: string              // "b2bua"
  readonly host: string
  readonly port: number
  readonly uriParams?: Record<string, string>  // { callRef, leg, emerg } — opaque to stack
}
```

### Generators (all pure)

```ts
// Out-of-dialog request: initial INVITE, one-shot OPTIONS.
generateOutOfDialogRequest(method: "INVITE" | "OPTIONS" | "MESSAGE" | ..., opts: {
  readonly requestUri: string
  readonly callId: string
  readonly fromUri: string
  readonly fromTag: string
  readonly toUri: string
  readonly toTag?: string              // normally absent
  readonly cseq: number
  readonly via: ViaSpec
  readonly contact: ContactSpec
  readonly maxForwards?: number        // defaults 70
  readonly body?: Uint8Array
  readonly contentType?: string        // required if body present
  readonly extraHeaders?: ReadonlyArray<SipHeader>
}): SipRequest

// In-dialog request: BYE, re-INVITE, PRACK, NOTIFY, INFO, UPDATE, MESSAGE, OPTIONS.
// NOT ACK — ACK is its own primitive (see below).
generateInDialogRequest(
  method: "BYE" | "INVITE" | "PRACK" | "NOTIFY" | "OPTIONS" | "INFO" | "UPDATE" | "MESSAGE",
  dialog: StackDialog,
  opts: {
    readonly via: ViaSpec
    readonly contact: ContactSpec
    readonly body?: Uint8Array
    readonly contentType?: string
    readonly extraHeaders?: ReadonlyArray<SipHeader>
    readonly rack?: string             // PRACK only
    readonly event?: string            // NOTIFY only
    readonly subscriptionState?: string // NOTIFY only
  }
): { readonly request: SipRequest; readonly dialog: StackDialog }   // dialog.localCSeq bumped

// ACK for 2xx — separate because CSeq number comes from the INVITE txn, not the dialog.
// PRACK between INVITE and 2xx causes dialog.localCSeq to no longer equal INVITE's CSeq.
generateAckFor2xx(
  inviteTxn: InviteClientTransactionHandle,
  dialog: StackDialog,
  opts: {
    readonly via: ViaSpec              // new branch — ACK for 2xx is its own hop
    readonly body?: Uint8Array
    readonly contentType?: string
    readonly extraHeaders?: ReadonlyArray<SipHeader>
  }
): SipRequest

// CANCEL — reuses INVITE's branch, extracts everything else from the INVITE.
generateCancel(inviteTxn: InviteClientTransactionHandle): SipRequest

// UAS response — echoes Via / From / To / Call-ID / CSeq from the request.
generateResponse(
  incomingRequest: SipRequest,
  status: number,
  reason: string,
  opts?: {
    readonly toTag?: string            // caller pins (first >100 response); reuse for later ones
    readonly contact?: ContactSpec     // REQUIRED on 2xx to dialog-creating requests (INVITE/SUBSCRIBE/…)
    readonly body?: Uint8Array
    readonly contentType?: string
    readonly extraHeaders?: ReadonlyArray<SipHeader>
  }
): SipResponse

// Transparent-header helper — extracts non-structural headers from an inbound
// message so the caller can pass them back via `extraHeaders` when relaying.
// Structural = Via, Contact, From, To, Call-ID, CSeq, Max-Forwards,
// Content-Length, Content-Type, Record-Route, Route.
extractNonStructuralHeaders(msg: SipMessage): ReadonlyArray<SipHeader>

// Stack-internal — called only by TransactionLayer auto-ACK machinery.
// Keeps the RFC §17.1.1.3 invariant inside the stack.
_generateAckForNon2xx(originalInvite: SipRequest, finalResponse: SipResponse): SipRequest
```

## RFC 3261 rules encoded by each generator

| Generator | RFC rules enforced |
|---|---|
| `generateOutOfDialogRequest` | §8.1.1: Via, Max-Forwards default 70, From-tag, Call-ID, CSeq, To, Contact. §13.2.1 for INVITE. |
| `generateInDialogRequest` | §12.2.1.1: Call-ID, dialog's From/To with swapped tags, `CSeq: localCSeq+1`, Request-URI = remote-target, route set. §7.4.1: Content-Type when body non-empty. |
| `generateAckFor2xx` | §13.2.2.4: CSeq number from INVITE (via handle), method "ACK". New Via branch (own hop). Route set + remote-target from dialog. §17.1.1.2: terminates INVITE client txn on send. |
| `generateCancel` | §9.1: Request-URI, Call-ID, From, To (no tag added beyond INVITE's), CSeq number from INVITE. Via branch **reused** from INVITE (mandated for server matching). |
| `generateResponse` | §8.2.6.2: echo Via, From, To, Call-ID, CSeq. Add To-tag when status > 100 and caller provides. §12.1.1: UAS tag pinned across all non-100 responses — caller's responsibility to pass the pinned value. §20.10: Contact required on 2xx to dialog-creating requests. |
| `_generateAckForNon2xx` | §17.1.1.3: reuse INVITE's Via (same branch), copy From/To/Call-ID, `CSeq: <inviteNumber> ACK`. |

## Slice plan

Each slice must end with `npm run typecheck && npm run test` green.

### Slice A.1 — Land new generators alongside old builders

**Scope:** `src/sip/generators.ts` new file with all six generators. `src/sip/Dialog.ts` new file with `StackDialog`. New tests under `tests/unit/sip/generators.test.ts`.

**Deliverables:**
- `generateOutOfDialogRequest`, `generateInDialogRequest`, `generateAckFor2xx`, `generateCancel`, `generateResponse`, `extractNonStructuralHeaders` implemented.
- Unit tests per generator: golden-input → golden-output byte match.
- **Byte-equivalence tests** against the existing builders for the same logical inputs — proves no semantic drift. Delete once old builders are removed.

**No call sites migrate in this slice.** Pure addition; risk is isolated to new files.

**Exit criteria:** `npm run test` green; byte-equivalence tests prove each new generator matches the corresponding old builder for a fixed corpus of inputs.

### Slice A.2 — ClientTransactionHandle return type

**Scope:** `src/sip/TransactionLayer.ts`. Split `send()` into `sendRequest` (returns handle) and `sendResponse` (returns void). Old `send()` becomes a deprecated thin wrapper mapping to one of the two.

**Deliverables:**
- `ClientTransactionHandle` type exported.
- `sendRequest` returns a handle containing the branch + `originalInvite` / `originalRequest` + destination.
- Tests: assert handle shape for INVITE and BYE sends.

**Exit criteria:** Existing tests green. No call site yet uses the handle.

### Slice B.1 — `StackDialog` composition

**Scope:** `src/sip/Dialog.ts` (new), `src/call/CallModel.ts` (Dialog becomes `{ sip, ext }`). Migrate every `dialog.X` read/write in the B2BUA to `dialog.sip.X` or `dialog.ext.X`.

**Churn:** ~40 call sites across [`ActionExecutor.ts`](../../src/b2bua/rules/framework/ActionExecutor.ts), [`DialogRules.ts`](../../src/b2bua/rules/defaults/DialogRules.ts), [`TransferRules.ts`](../../src/b2bua/rules/defaults/TransferRules.ts), [`CornerCaseRules.ts`](../../src/b2bua/rules/defaults/CornerCaseRules.ts), [`LifecycleRules.ts`](../../src/b2bua/rules/defaults/LifecycleRules.ts), [`TerminatingRules.ts`](../../src/b2bua/rules/defaults/TerminatingRules.ts), [`RelayRules.ts`](../../src/b2bua/rules/defaults/RelayRules.ts), [`FailureRules.ts`](../../src/b2bua/rules/defaults/FailureRules.ts), [`relayFirst18xTo180.ts`](../../src/b2bua/rules/custom/relayFirst18xTo180.ts), [`SipRouter.ts`](../../src/sip/SipRouter.ts).

**Exit criteria:** All existing e2e tests in [`tests/fullcall/`](../../tests/fullcall/) pass. Typecheck clean.

**Risk:** Highest-churn slice. Mitigation: the change is mechanical (field renames), and TypeScript forces every site to be touched. No behaviour change — only data-path rename.

### Slice B.2 — Introduce `pendingInviteTxn` fields

**Scope:** Add `pendingInviteTxn?: InviteClientTransactionHandle` to `Leg` and `B2buaDialogExt`. Populate at send sites. Old duplicate state (`leg.inviteBranch`, `dialog.ext.lastInviteCSeq`) stays read-only, consulted by old code paths.

**Exit criteria:** Handle flows alongside duplicates; no behaviour change yet.

### Slice C.1 — Migrate CANCEL to `generateCancel(handle)`

**Scope:** [`ActionExecutor.buildCancelEnvelope`](../../src/b2bua/rules/framework/ActionExecutor.ts#L171) and all CANCEL call sites (~3 sites). Delete `leg.inviteBranch` writes; reads move to `leg.pendingInviteTxn?.branch` only for any residual consumer (should be zero after migration).

**Test:** existing e2e CANCEL scenarios must stay green. Branch-reuse invariant (§9.1) is now enforced inside `generateCancel` rather than via `forceBranch` on `stampHeaders`.

**Exit criteria:** `leg.inviteBranch` field deleted in this slice; typecheck green.

### Slice C.2 — Migrate ACK-for-2xx to `generateAckFor2xx(handle, dialog)`

**Scope:** [`ActionExecutor.executeAckLeg`](../../src/b2bua/rules/framework/ActionExecutor.ts#L888) and related ACK call sites. Remove `dialog.ext.lastInviteCSeq` reads. Delete the field once no reader remains.

**Tests:**
- **Critical:** a PRACK-between-INVITE-and-2xx e2e scenario must exist and stay green. If absent in [`tests/fullcall/`](../../tests/fullcall/), add one before this slice lands.
- Forking-with-multiple-early-dialogs e2e stays green (ACK-to-winner with INVITE's CSeq).

**Exit criteria:** `lastInviteCSeq` field removed; ACK CSeq source is exclusively the txn handle.

### Slice C.3 — Migrate in-dialog requests

**Scope:** [`ActionExecutor.ts`](../../src/b2bua/rules/framework/ActionExecutor.ts) — all `buildBye` / `buildRelayedBye` / `buildOriginatedInvite` / `buildOptions` / `buildPrack` / `buildRelayedPrack` / `buildRelayedRequest` call sites.

All `dialog.localCSeq + 1` arithmetic deleted — the generator returns the updated dialog.

`copyFrom` removed; callers use `extractNonStructuralHeaders(inboundMsg)` and pass the result via `extraHeaders`.

**Tests:** Full [`tests/fullcall/`](../../tests/fullcall/) e2e suite. Anomaly scan via `/sip-callflow-review`.

**Exit criteria:** zero direct `MessageFactory.build{Bye,Prack,Options,Originated,Relayed*}` imports in `src/b2bua/`.

### Slice C.4 — Migrate responses

**Scope:**
- [`SipRouter.ts`](../../src/sip/SipRouter.ts) unroutable-481 and vanished-call-481 paths (lines 625, 653).
- [`B2buaCore.ts:82`](../../src/b2bua/B2buaCore.ts#L82) unsupported-method 501.
- [`InitialInviteHandler.ts`](../../src/b2bua/InitialInviteHandler.ts) 503/code/486 rejects (lines 74, 92, 187).
- [`ActionExecutor.ts:339`](../../src/b2bua/rules/framework/ActionExecutor.ts#L339) `rejectResponse`.
- [`ActionExecutor.relayResponseMsg`](../../src/b2bua/rules/framework/ActionExecutor.ts#L591) b-leg→a-leg response relay. Uses `generateResponse(aLegInvite, status, reason, { toTag, contact, body, extraHeaders: extractNonStructural(bLegResp) })`.
- [`TransactionLayer.ts:419`](../../src/sip/TransactionLayer.ts#L419) auto-200 to CANCEL and line 424 auto-487 — move to `generateResponse(req, 200, "OK", { toTag: uasToTag })` and `generateResponse(invite, 487, "Request Terminated", { toTag: uasToTag })`.
- [`TransactionLayer.ts:511`](../../src/sip/TransactionLayer.ts#L511) `build100Trying` → `generateResponse(req, 100, "Trying")`.

**Note:** This slice requires the B2BUA to retain the original a-leg INVITE for generateResponse in the response-relay path. Today [`Call`](../../src/call/CallModel.ts) already retains `aLegVias` / `aLegFrom` / `aLegTo` / `aLegInviteCSeq` — retain the full `SipRequest` instead and delete those four denormalised fields.

**Tests:** e2e suite. Callflow-review must show no drift on rejection / relay scenarios.

**Exit criteria:** All `MessageFactory.build{200Ok,487,491,100Trying,RejectResponse,relayResponse}` imports gone.

### Slice C.5 — Migrate initial INVITE

**Scope:** [`b2bua/helpers.ts:189 buildBLegInvite`](../../src/b2bua/helpers.ts#L189) → `generateOutOfDialogRequest("INVITE", …)` + `extractNonStructuralHeaders(aLegInvite)` for the transparent-header copy.

**Tests:** full call e2e suite.

**Exit criteria:** `buildBLegInvite` deleted.

### Slice C.6 — Migrate TransactionLayer auto-ACK for non-2xx

**Scope:** [`TransactionLayer.buildAckForNon2xx`](../../src/sip/TransactionLayer.ts#L150) (local duplicate) → `_generateAckForNon2xx(originalInvite, response)` exported from generators for stack-internal use only.

**Tests:** 3xx/4xx/5xx failure e2e scenarios.

**Exit criteria:** local `buildAckForNon2xx` deleted; single ACK-for-non-2xx implementation.

### Slice D.1 — Delete `__PLACEHOLDER__` and `stampHeaders`

**Scope:** [`SipRouter.stampHeaders`](../../src/sip/SipRouter.ts#L197) deleted. `SipRouter.processResult` becomes a simple serialize-and-send loop. All Via/Contact branch-capture logic (lines 324–368) disappears because branches are now generated by generator callers and persisted into `pendingInviteTxn` / `ackBranch` at call-site time.

**Tests:** full suite.

**Exit criteria:** no `"__PLACEHOLDER__"` string anywhere in `src/`.

### Slice D.2 — Retire old builders

**Scope:** Delete all `build*` functions from `src/sip/MessageFactory.ts`. Rename remaining helpers (`getHeader`, `getHeaders`, `setHeader`, `removeHeader`, `extractTag`, `stripTag`, `extractNameAddrUri`, `extractContactUri`, `parseSipUri`, `extractHostPort`, `parseUriParams`, `parseViaParams`, `newCallId`, `newTag`, `newBranch`, `isEmergencyRequest`, `bufferHasEmergencyMarker`, `bufferHasToTag`, `isInviteRequestBuffer`, `buildStatelessReject503Buffer`) into `src/sip/MessageHelpers.ts`.

**Note:** `buildStatelessReject503Buffer` stays as-is — byte-slicing without a parsed request is a legitimate stack primitive for the Tier-1 dispatcher and doesn't overlap with the generators.

**Exit criteria:** `MessageFactory.ts` file removed from the tree.

## Testing strategy

### Unit tests (added in A.1)

Per generator, under `tests/unit/sip/generators.test.ts`:
- Golden inputs → golden serialized bytes.
- **Byte-equivalence to old builder** for the same logical inputs — kept until the old builder is deleted.
- Edge cases:
  - `generateInDialogRequest` rejects `method === "ACK"` at the type level.
  - `generateAckFor2xx` reads CSeq from handle, not dialog.
  - `generateCancel` reuses INVITE branch verbatim.
  - `generateResponse` echoes Via / From / To / Call-ID / CSeq from the request.
  - `extractNonStructuralHeaders` filters the full structural list.
  - URL-encoding of `customParams` / `uriParams` values with `;`, `=`, `@`, spaces.

### Regression via existing e2e

[`tests/fullcall/`](../../tests/fullcall/) covers full-call, REFER, transfer, CANCEL, PRACK-absorption, failover (relayFirst18xTo180). Every C-slice must run the full e2e suite and the `/sip-callflow-review` anomaly scan with no new diffs.

### Dedicated pre-slice-C.2 additions

Before C.2 lands:
1. Verify a **PRACK-between-INVITE-and-2xx** e2e case exists in the suite. If not, author one. This is the regression surface for removing `lastInviteCSeq`.
2. Verify a **forking with multiple early dialogs** e2e case exists. Author one if not.

### Byte-level snapshot comparison

During slice C.* migrations, capture the serialized wire bytes of every outbound message across a fixed e2e scenario set before and after each slice. Any difference that isn't an intended structural change (e.g. `Max-Forwards` order) is a regression.

## Out of scope

- PCAP replay tooling (separate TODO).
- Redis / cache schema changes to carry the new `pendingInviteTxn` through restarts — handles live only in-memory, exactly like today's `leg.inviteBranch`.
- Rule framework ADT changes (independent refactor in [`RULE-FRAMEWORK-ADT-REFACTOR.md`](RULE-FRAMEWORK-ADT-REFACTOR.md)).
- Changing the transparent/structural partition — stays as RFC 3261 §16.6 defines.
- Removing `buildStatelessReject503Buffer` — byte-slicing overload-protection path is legitimately different from the generator family.

---

## Open risks

1. **Response-relay retention change** (slice C.4). Switching from denormalised `aLegVias` / `aLegFrom` / `aLegTo` / `aLegInviteCSeq` to "retain the full `SipRequest`" slightly raises per-call memory footprint. Measure before / after on a sustained-load run.
2. **Handle lifetime on `leg.pendingInviteTxn`.** Must be cleared once ACK is sent (initial INVITE) or the INVITE txn terminates (on final response for non-2xx). A leaked handle keeps a parsed `SipRequest` alive per b-leg. Assert cleanup in a dedicated unit test for each terminal path.
3. **`dialog.ext.pendingInviteTxn` on glare / 491.** Re-INVITE that receives 491 never ACKs (no 2xx) — handle must be cleared on 491 receipt too. Cover in a C.2 test.
4. **Forking with shared INVITE handle.** Multiple early dialogs on one leg share `leg.pendingInviteTxn`. ACK-for-2xx reads it; after the first ACK-for-2xx, siblings are killed via BYE. Ensure the handle isn't cleared until all forks are resolved. Covered by the forking e2e added pre-C.2.
