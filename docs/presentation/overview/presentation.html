SIP B2BUA

Distributed Telephony · Effect-TS · Redis Replication

Architecture · SIP Stack Behavior · Call Pinning · State Replication

What it is

SIP Back-to-Back User Agent — a programmable telephony switch that terminates and re-originates SIP dialogs.

  • Listens on UDP SIP (port 5060)
  • Calls an HTTP backend to decide call routing
  • Bridges two legs (a-leg, b-leg) with full header rewriting
  • Operates as a stateful B2BUA, not a stateless proxy

Two deployment modes

Standalone production server, or embedded library with custom CallDecisionEngine

Multi-agent test framework

In-process registrar drives alice.register() / bob.register() scenarios against real PBX/SBC

Built on Effect v4

Typed errors, structured concurrency, deterministic TestClock testing

The problem it solves

A B2BUA hides Alice from Bob (and Bob from Alice) so the operator can impose policy between them — routing, header rewriting, billing, failover, regulatory controls.

Routing

Per-call destination chosen by HTTP backend on every POST /call/new

Failover

Multiple b-legs per call: b-1, b-2… probe in sequence on no-answer / SIP failure

Isolation

Independent CSeq, tag, Call-ID spaces. Each leg sees only its peer's surface

Limiters

Per-customer windowed call counters, check-and-increment on INVITE

Emergency

Resource-Priority: esnet / wps / q735 → bypass shedding, fast-path through overload tiers

Survival

Worker death does not drop in-flight calls — backup serves the dialog

Key architectural decisions

1

Stateless proxy, stateful workers

Edge owns the UDP socket and routing decision. All call state lives on workers. Proxy can be killed and respawned without consulting any peer.

2

HMAC Record-Route cookie for call pinning

In-dialog traffic is bound to its worker by a signed cookie inserted at INVITE time. No proxy-side routing table, no per-call lookup. Survives proxy restart.

3

Rendezvous (HRW) hashing, not modular

On worker pool change, HRW moves only 1/N of calls. Modular hashing reshuffles (N-1)/N.

4

Two-Redis topology — sidecar + cluster

Per-pod sidecar for hot-path call state (localhost latency). Shared Redis only for CallLimiter (global counters need consensus, latency-tolerant).

5

INAP-inspired declarative rule framework

All SIP behavior expressed as rules returning typed actions. ActionExecutor enforces SIP invariants. Rules never write SIP messages directly.

6

Single-owner invariant on call partition

A call's primary worker is fixed at INVITE time and never reassigned. Backups serve and replicate back, but the partition reference stays with the primary.

Two distinct Redis topologies

Different access patterns demand different placements. Mixing them was an early design mistake.

Per-pod sidecar Redis

HOT PATH

  • Co-located with each B2BUA worker (same pod)
  • UNIX socket / localhost — sub-millisecond writes
  • Holds all call context: pri:N:call:*, bak:P:call:*, idx:*, propagate:*, epoch:N
  • Worker is the only writer to its own sidecar
  • Crash = sidecar wipes with the pod; recovery pulls from peers

Cluster-shared Redis

CONSENSUS

  • Single shared instance across all workers
  • Holds CallLimiter only — windowed customer counters
  • Needs global view; one worker per call is not enough
  • Latency-tolerant — bounded check-and-increment, not on every packet
  • Loss = transient limiter inaccuracy, not call loss

Rule of thumb: data that must be local for SIP timing → sidecar. Data that must be global for correctness → shared.

SIP Stack Behavior

Call / Leg / Dialog · Header rewriting · Rule framework

Call / Leg / Dialog hierarchy

Why three levels?

  • Call — the operator concept. One CDR, one limiter charge, one routing decision.
  • Leg — the SIP transaction concept. One Call-ID and From-tag per leg; failover spawns b-2.
  • Dialog — the RFC 3261 dialog. Multiple per leg during forking; one survives.

Every framework primitive (merge, split, destroy-leg, add-tag-mapping) operates at exactly one of these levels.

State invariants

  • trying: INVITE sent, no provisional with tag
  • early: 1xx with To-tag, no 2xx
  • confirmed: 2xx received, ACK sent
  • terminated: BYE exchanged, timer expired, or failure response

Disposition (independent axis)

  • pending · bridged · cancelling · rejected

Tag isolation & dialog correlation

Alice and Bob live in independent dialog spaces. The B2BUA bridges them via a translation table.

What this buys

  • Alice cannot drive Bob's transaction state directly
  • Forking: b-1, b-2 add entries; only the bridged one's aTag reaches Alice
  • Re-INVITE/UPDATE relay is just a tag-table lookup

Where it's enforced

  • add-tag-mapping primitive on dialog confirmation
  • activePeer structurally enforces 1↔1 peering
  • InvariantEnforcer rejects rule outputs that break dialog isolation

SIP Front Proxy

Stateless edge · HMAC call pinning · Shared VIP HA

Proxy responsibilities

Inbound — UAC side

  • Owns UDP :5060 socket (shared VIP)
  • Distributes initial INVITEs via HRW hashing on Call-ID
  • Inserts HMAC-signed Record-Route cookie
  • Decodes cookie on in-dialog requests, forwards to pinned worker
  • CANCEL correlation via TTL'd LRU on (Call-ID, CSeq)

Outbound — UAS side

  • Forwards worker-originated requests to peer
  • Adds its own top Via with own branch — replies route back
  • Source-IP discipline: forwarded SIP uses VIP, OPTIONS health probes use per-proxy node IP

What it never does

  • Hold per-call state — restart loses zero calls
  • Consult a routing table — cookie is self-describing
  • Coordinate with the other proxy — both probe workers independently

Call pinning — Rendezvous (HRW) hashing

rendezvousSelect(callId, workers): best = null bestScore = 0 // Healthy peers only for w in workers where w.health == 'alive': key = callId + ":" + w.id digest = SHA-1(key) score = uint64(digest[0..8]) // top 8 bytes if score > bestScore: bestScore = score best = w return best

Why not fnv1a(callId) % N?

  • Modular: adding/removing a worker reshards (N-1)/N of calls
  • HRW: only 1/N of calls move on membership change
  • Critical when scaling pool from 3→4 nodes during traffic

Pool stability

  • Workers are a StatefulSet — pod ordinals worker-0..N survive restarts
  • Deployment's random suffixes would invalidate every cookie
  • HRW scoring depends only on stable worker.id

The HMAC stickiness cookie

Encoded into the Record-Route URI: Record-Route: <sip:proxy:5060;w=worker-3;v=2;w_pri=3;w_bak=7;kid=a3f1...;sig=K7n…;lr>
ParamMeaning
wWorker pod name (StatefulSet ordinal, e.g. worker-3)
vCookie format version (2 includes w_bak)
w_pri/w_bakPrimary & backup ordinals — fixed at INVITE, never reassigned
kidHMAC key id — first 16 hex of SHA-1(keyBytes); supports rotation
sigHMAC-SHA256 truncated to 128 bits (RFC 4868 §2.6), base64url

Why HMAC?

Cookie is sent back by Alice/Bob. Without signing, a malicious UA could rewrite w= and redirect another customer's call.

Key rotation

Verify against (current, previous) pair. 1h overlap allows safe rotation without dropping in-flight dialogs.

B2BUA Worker

Pipeline · Rule framework · Action primitives

Worker pipeline

Per-call serialization

  • One Effect.Semaphore(1) per callRef
  • Rule matching, mutation, Redis flush, outbound send all run sequentially per call
  • Proxy guarantees affinity → no distributed lock needed
  • Different calls run on different fibers → unbounded concurrency across calls

Layer responsibilities

  • TransactionLayer — RFC 3261 client/server transactions, retransmission
  • RuleExecutor — specificity-scored rule dispatch, custom-rule overlay
  • ActionExecutor — only thing that mutates call state & emits SIP
  • InvariantEnforcer — catches rules trying to break SIP semantics

docs/AdvancedCallModel.md:18–34

The rule framework

All call processing is declarative: a rule is a match descriptor plus an effect that returns typed actions. The framework owns SIP semantics; rules own business decisions.

interface Rule { name: string match: Match // discriminated union by kind: // request | response | timer | timeout | cancelled | internal-event overrides?: string[] composesWith?: string[] run: (ctx: CallContext) => Effect<Action[], CallError, Deps> }

Specificity-scored dispatch

  • Each rule's match contributes a score for incoming event
  • Highest score wins. Ties → registration error unless overrides: or composesWith: declared
  • Forces explicit conflict resolution at registration time, not at runtime

Built-in vs custom

  • Default lifecycle rules: relay-options, relay-bye, relay-ack, handle-cancel, route-failure, no-answer-failover, max-duration
  • Corner cases: cancel-200-crossing, retransmit-200, reinvite-glare (491)
  • Custom rules: activated per-call via HTTP /call/new response — REFER, 183-to-200, MRF planned

Replication

Single-owner invariant · Dual-Redis backup · /replog protocol

Why replicate? The failure model

What we can't do

  • Cross-pod Redis replication — too slow on SIP hot path; INVITE → 200 OK is < 200ms, can't wait for cross-AZ ack
  • Shared cluster Redis for call state — single-point latency, contention, blast radius
  • Cold rehydrate from CDR — CDRs are append-only, missing live state (CSeq, dialog routes)

What we chose

  • Sidecar-to-sidecar pull replication
  • Primary writes locally (sub-ms), pulls happen async over HTTP
  • Per-call (primary, backup) pair fixed at INVITE time
  • Backup keeps a shadow body, serves if primary dies
  • Recovery: replays from peer logs at boot

Critical timing constraint

Primary restart must complete within callContextTtlSec (default 30 min = keepalive × 2). After TTL, the backup body expires — call lost on next message.

The single-owner invariant

A call's primary owner is fixed at INVITE time and never changes for the call's lifetime. docs/replication/call-cache-backup.md:20–46

What this means

  • Proxy stamps w_pri and w_bak into the cookie at INVITE time
  • Those ordinals travel with the call from INVITE through BYE, regardless of which worker physically handled any message
  • pri:{P}:call:{ref} is written only by worker P, ever

Why?

  • Avoids partition-flip races (two workers both think they own a call)
  • Reverse-propagate is a single, well-defined direction
  • "Who is authoritative?" is a constant-time lookup

"Backup never promotes" — what it really means

NOT backup refuses to serve

YES backup serves traffic, bumps callGen, writes new state to bak:{primary}:call:{ref} on its own sidecar

YES updates flow back via propagate:{B}->{A} so primary recovers them on return

NEVER backup moves the call body into pri:{B}: — partition reference stays with A forever

Storage layout per worker sidecar

Each worker N's sidecar Redis holds these namespaces:

KeyContents
pri:{N}:call:{ref} JSON body for calls where N is primary · carries callGen, written_at_ms, _topology={wPri,wBak}, full Call shape
bak:{P}:call:{ref} JSON body for calls where N backs up primary P · same shape, one entry per peer primary
idx:{indexKey} → callRef SIP-derived secondary index (Call-ID, dialog tags) · pure function of call body · never on the wire
propagate:{N}→{P}:gen:{entryGen} Sorted set per channel · members U:{bodyKey} / D:{bodyKey} · entryGen=0 mirror, =g originating
seq:{N}→{P}:gen:{entryGen} Per-bucket counter, INCR'd on each write
epoch:N Worker boot counter (K8s downward API restartCount + boot-ms fallback)
replpos:{peer} Puller bookkeeping per peer: { epoch, lastSeq }

Invariant: indexes are derived from the body — every receiver re-derives via callIndexKeysFromUnknown. Wire protocol is index-schema-agnostic.

Replication topology — write & pull paths

Cycle warning: A's originating write goes to B → B's puller mirrors it back into propagate:B→A with entryGen=0 → A's puller rejects (mirror entries excluded for warm pullers, included only for cold recovery).

The wire protocol — /replog long-poll

GET /replog?caller=worker-A&gen=42&counter=15&chunk_size=64 → application/x-ndjson, streamed DataFrame (one per replicated mutation): NoopFrame (heartbeat + caught-up signal): { { "type": "noop", "type": "data", "gen": 42, "counter": 15, "gen": 42, "counter": 16, "latency_ms": 0 } "op": "update", // create|update|delete "partition": "pri", // flipped on apply "callRef": "abc123", "body": { "callGen": 5, "written_at_ms": …, "_topology": {…}, … }, "body_ttl_remaining_sec": 540, "latency_ms": 12 }

Long-poll semantics

  • Puller opens a stream per peer; server holds it open
  • Each new write → server emits a DataFrame
  • Idle → periodic NoopFrame proves liveness, advances watermark
  • First Noop flips everCaughtUp = true → WorkerReadiness.markReady(true)

Watermark & resumption

  • Puller persists {epoch, lastSeq} in replpos:{peer}
  • Reconnect sends last-seen (gen, counter)
  • Server walks propagate sets from that position
  • Cold start = (0, 0) → full backfill

The cycle-break — entryGen = 0 sentinel

The naïve cycle: A writes → B applies + writes outgoing-to-A → A pulls → re-applies → forever.

Two kinds of entries live in propagate: Originating writes entryGen = self.incarnationGen (e.g. 42) ← real call mutations Mirror writes entryGen = 0 ← entries the puller created when applying What the puller does when receiving a frame: apply(frame): targetPartition = frame.partition == "pri" ? "bak" : "pri" // PARTITION FLIP targetOwner = (targetPartition == "bak") ? frame.source : self kv.channelWriteUpdate({ entryGen: 0, // always mirror — never my own incarnationGen channel: `propagate:{self}->{frame.source}`, bodyKey: `${targetPartition}:${targetOwner}:call:${frame.callRef}`, bodyValue: serializeBody(frame.body), indexes: callIndexKeysFromUnknown(frame.body).map(…) }) When A pulls B's outgoing-to-A with watermark (B.incarnationGen, lastCounter): Bucket gen=0: skipped — all (0, *) ≤ A's watermark ✓ cycle broken Bucket gen=B.incarnationGen: contains only B's originating writes ← A applies these

Same mechanism powers cold recovery. Fresh A boots with watermark (0, 0) → bucket gen=0 is now > watermark → puller pulls ALL mirrors → repopulates pri:{A}: from B's shadow copies.

Failover scenarios & the callGen gate

Per-call content gate

Wire (gen, counter) isn't enough — cross-direction writes on the same call need a content-level tiebreaker.

apply(frame): local = kv.bodyGet(targetBodyKey) localGen = local ? parseCallGen(local) : -∞ if frame.body.callGen ≤ localGen: skip else: write
  • Bumped only by primary-originating writes (RMW)
  • Puller preserves callGen verbatim
  • Cold recovery (-∞) accepts any incoming

Overload Protection

Three-tier defence · Byte brake · Token bucket · Emergency bypass

Three-tier defence

Order matters. Tier 1 protects parser CPU. Tier 2 protects worker fairness across classes. Tier 3 protects against tail-latency cascades.

Emergency call propagation

Detection

  • Initial INVITE carries Resource-Priority: esnet.0|wps.0|q735.0
  • InitialInviteHandler sets call.emergency = true
  • SipRouter stamps ;emerg=1 on outbound Contact, ;em=1 on Via

Why stamp on every outbound?

  • In-dialog packets must be recognizable by Tier 1's byte-scan
  • Tier 1 sees ;em=1 in the buffer → bypass 503 template
  • No JsSIP parse needed to make the priority decision

Recap — what makes this system work

Separation of concerns

  • Stateless edge for routing & pinning
  • Stateful workers for SIP semantics
  • Declarative rules for call behavior
  • Two Redis topologies for two access patterns

Discipline of invariants

  • Single-owner partition reference
  • 1↔1 leg peering via activePeer
  • Reach equals parameters for primitive actions
  • Indexes derived, never sent on wire

What replication buys

  • Worker death does not drop in-flight calls
  • Sidecar wipe recovers from peers within 30s P99
  • Backups serve traffic, never split-brain primary
  • Wire protocol is index-schema-agnostic

What's accepted lost

  • Both primary & backup simultaneously dead (AZ-level)
  • Primary restart longer than callContextTtl (~30 min)
  • In-flight tx timers on a dying proxy
