# `decode_forward` to a respawned-NotReady primary returns 481 — fix plan

## TL;DR

The proxy-drain test (`tests/k8s/proxy-drain.test.ts`) fails ~40% of runs with
`8/20–10/20 calls succeeded`. We traced the failure end-to-end via:

1. **sipp screen log** — `BYE → 200 OK 10/20`, `Unexpected-Msg 10/20`. All
   20 BYEs are sent on the wire; sipp gets 200 OK back for only the
   survivor's local calls.
2. **sipp `msg.log`** counted by status — `200 OK = 10`,
   `481 Call/Transaction Does Not Exist = 38` (10 distinct calls × ~4
   retransmits each).
3. **Survivor redis** — `bak:* count == drained-call count`,
   `idx:leg:<callId>|<fromTag>` is present for every migrated call.
   Indexes are NOT missing; bak: bodies have `state="active"`.
4. **Proxy decision log** — for every drained-pod-bound BYE, the
   recorded decision is `decode_forward` to the **new pod IP** of the
   respawned drained worker (not `decode_forward_backup` to the
   survivor).

So neither the bak: index replication nor the worker-side `checkout`
fallback is at fault. The 481 is generated by the **respawned drained
pod** because its Redis sidecar is empty (started fresh on respawn) and
the proxy is sending the BYE to it instead of to the survivor that holds
the bak: copy.

## Root cause (one sentence)

**The proxy's `decode_forward` resolves the cookie's `w_pri` ordinal to
the current pod IP without consulting whether that pod is `Ready`,
which means a respawned-but-still-draining-replication primary receives
in-dialog traffic for calls it knows nothing about and returns 481.**

Slice D ([d9-readiness](../replication/call-cache-backup.md#d9))
correctly keeps the new pod **out of K8s `Endpoints`** until
`ReadyGate` finishes draining peers. K8s `Endpoints` and SIP `OPTIONS`
both advertise the pod as "not ready" during this window. But the proxy's
**decode_forward path** uses an *ordinal → pod-IP* lookup that bypasses
the readiness gate — the moment the StatefulSet recreates a pod, that pod
appears in the `PodInformer` view and decode_forward happily targets it,
regardless of `/ready` or OPTIONS-503.

## The five-step failure timeline (concrete numbers from molxec6v)

```
t=0s       sipp launches 20 INVITEs (rate 20/s).
            12 land on b2bua-worker-1 (drained), 8 on b2bua-worker-0 (survivor).
            All 20 establish (180 → 200 OK → ACK).
t≈10–12s   Test issues `kubectl delete pod b2bua-worker-1
            --grace-period=30 --wait=false`.
            Worker process receives SIGTERM.
            DrainingState flips to draining.
            Process exits in <2s (no preStop hook in test values).
            Pod is in `Terminating` phase but K8s grace window is 30s.
t≈12–18s   StatefulSet controller recreates pod b2bua-worker-1
            (new IP, fresh Redis sidecar via emptyDir).
            New worker process starts. Initial state:
              - WorkerReadiness.currentReady = false
              - DrainingState.mode = "serving"
              - pri:b2bua-worker-1:* = ∅  (sidecar empty)
              - bak:* = ∅
            ReadyGate begins draining propagate:b2bua-worker-1 from
            survivor (b2bua-worker-0). Drain takes a few hundred ms in
            normal cases; the worker stays NotReady for the duration.
t≈25s      sipp emits BYEs (15s after ACK per uac-hold-failover.xml).
            Each BYE traverses the proxy.
            Proxy receives BYE for cookie {w_pri: b2bua-worker-1,
            w_bak: b2bua-worker-0, ...}.
            decode_forward path:
              1. Verify cookie signature → OK.
              2. Resolve w_pri ordinal → current pod IP via PodInformer.
              3. PodInformer has the new pod (Phase=Running, Ready=false).
              4. Forward BYE to that IP.
            BYE arrives at the new b2bua-worker-1.
            Its resolveFromSipKey misses (idx:leg empty).
            SipRouter generates 481 "Call/Transaction Does Not Exist".
            sipp counts it as Unexpected-Msg.
            The 12 calls that were on the drained primary fail.
```

## Why Slice D didn't cover this

Slice D set `WorkerReadiness=false` until `ReadyGate` completes, plumbed
that into the K8s `httpGet /ready` probe, AND made the SIP OPTIONS
keepalive return `503 (not-ready)` when WorkerReadiness is false. Both
gates work — verifiable by `kubectl get endpoints sip-front-proxy -o
yaml` showing the pod absent until ready.

But the proxy's `decode_forward` path **does not consult either signal**.
It uses a separate, ordinal-keyed view that resolves directly from the
PodInformer's pod list. We never extended decode_forward to honour the
readiness signals that Slice D produces.

## What "the right fix" looks like (decision matrix)

For an in-dialog request whose cookie carries `w_pri = X, w_bak = Y`:

| `w_pri` (X) state                | Today's decision     | Correct decision                        |
|----------------------------------|----------------------|-----------------------------------------|
| alive **and ready**              | decode_forward → X   | decode_forward → X                      |
| alive **and not-ready**          | decode_forward → X   | **decode_forward_backup → Y**           |
| draining (pre-grace, alive)      | decode_forward → X   | decode_forward → X                      |
| draining (post-grace, alive)     | decode_forward_backup→ Y | decode_forward_backup → Y           |
| dead / unknown / pod gone        | decode_forward_backup→ Y | decode_forward_backup → Y           |
| respawned, ReadyGate in flight   | decode_forward → X' (NEW POD IP, empty state) | **decode_forward_backup → Y** |

The new "alive but not-ready" row is what's missing. It's the row that
applies to a respawned primary mid-ReadyGate and to any future cause of
worker-side NotReady (e.g. soft-restart, large GC pause >readiness
timeout, etc.).

## Implementation plan (code-level)

### Slice E1 — Proxy honours per-worker readiness in decode_forward

Files (all under `src/sip-front-proxy/`):

1. **`health/HealthProbe.ts`** — extend `WorkerHealth` from
   `"alive" | "draining" | "dead" | "unknown"` to add `"not-ready"`.
   Distinguish via the OPTIONS reply already differentiated by my
   recent SipRouter change:
     - 200 OK + serving → `alive`
     - 503 with `Retry-After: 0` and worker-supplied reason `draining` → `draining`
     - 503 with `Retry-After: 0` and worker-supplied reason `not-ready (boot drain)` → `not-ready`
     - send timeout → `dead`
   Encoding the reason in the response: either parse the `Reason` header
   we add, or use distinct response phrases (`Service Unavailable
   (draining)` vs `Service Unavailable (not-ready)`). Reason header is
   cleaner.

2. **`strategies/LoadBalancer.ts`** — `decodeStickiness` matrix:
   add `not-ready → tryBackup` row. One-line change in the existing
   switch. Equivalent semantics to `dead`, but kept distinct so traces
   and metrics can tell them apart.

3. **`watch/PodInformer.ts`** — already exposes pod Ready condition;
   confirm it surfaces it through the consumer API. If not, add a
   `currentReady: boolean` field on the per-pod view. Don't change the
   set of pods the informer tracks — readiness is a side channel, not
   a membership filter.

4. **`ProxyCore.ts`** — when `decode_forward` would target ordinal X
   AND PodInformer reports `X.currentReady === false`, **promote the
   decision to `decode_forward_backup`** before sending. Single
   conditional, no new code path.

### Slice E2 — Metric + log breadcrumbs for the new path

Add a Prometheus counter `sipfp_decode_forward_promoted_total{from="not-ready"}`
and an INFO log when decode_forward is promoted to decode_forward_backup
because of NotReady. Without this, the failure mode is invisible to
operators (the cookie is valid, the ordinal exists, the pod is Running —
nothing looks wrong).

### Slice E3 — Conservative "stale Endpoints view" guard

The PodInformer's view of Ready is eventually consistent with K8s. If
the proxy hasn't yet observed the new pod's first probe failure, the
informer may report `Ready=true` for a pod that will fail the next
probe. To avoid that race, treat any pod whose `creationTimestamp` is
younger than `2 × probePeriod + initialDelay` as `not-ready` for routing
purposes regardless of the informer's current opinion. This is a small
upper-bound conservatism that closes the window between pod birth and
first probe success.

## How to test this WITHOUT the full K8s cluster

The failure path is genuinely SIP-end-to-end (cookie signed, decoded,
re-decoded, routed, BYE delivered to a worker that lacks state, 481
returned). But none of that requires real UDP sockets, real Redis, real
kubelet probes, or kind. The fake stack already simulates everything we
need:

  - `tests/support/fakeStack.ts` builds a worker with a memory
    `PartitionedRelayStorage` and a `SignalingNetwork.simulated`.
  - `tests/support/networkLeaves.ts` provides `WorkerReadiness.test(...)`
    and `DrainingState.test`. The boolean argument controls initial
    readiness.
  - The proxy can run on the simulated SignalingNetwork too (see
    `tests/support/proxyStack.ts` if it exists, or compose
    `ProxyCore` + simulated network directly).

We will add three fake-clock test files under `tests/sip-front-proxy/`:

### Test 1 — `decode-forward-not-ready.test.ts`

Setup:
  - Two workers, A (primary for the call) and B (backup).
  - One proxy in front, both workers visible via mock `PodInformer`.
  - A creates a call; cookie signed; replication seeds B's bak: copy.
  - Mark A's `WorkerReadiness` false (simulating mid-ReadyGate state).
  - PodInformer view of A: `Ready=false`.
  - Send BYE through the proxy (sipp is replaced by a `MessageBuilder`
    helper that constructs the BYE with the signed cookie inline).

Expectations:
  - Proxy decision MUST be `decode_forward_backup` (NOT `decode_forward`).
  - BYE MUST arrive at B (not A).
  - B MUST reply 200 OK (because B has the bak: copy).
  - sipp-side observer (a mock UAC fiber) MUST receive 200 OK.

This exact same test structure with `Ready=true` is the "control" — it
should result in `decode_forward → A → 200 OK from A`. Both directions
in one file makes the matrix obvious.

### Test 2 — `options-readiness-distinct.test.ts`

The proxy uses `OPTIONS` keepalive replies to classify worker health.
This test verifies the new four-way encoding:
  - serving + ready → `200 OK`
  - serving + not-ready → `503 + Reason: not-ready (boot drain)` (new)
  - draining + (any) → `503 + Reason: draining`
  - send timeout → no reply (proxy classifies `dead` after retry budget).

Pure SipRouter unit test. Drives `WorkerReadiness.markReady(false)` and
`DrainingState.markDraining` and inspects the response message produced
by the OPTIONS handler. No proxy involvement.

### Test 3 — `decode-forward-respawn-window.test.ts`

The "stale Endpoints" guard from Slice E3. Bring a fresh worker into
the PodInformer's view (creationTimestamp = now). Even if the informer
hasn't yet observed a probe outcome (so `Ready=undefined`), the proxy
must treat decode_forward against this worker as `not-ready` for the
guard window (`2 × probePeriod + initialDelay`). Advance the TestClock
past the window with no probe arriving, and the guard should still
hold (because no positive Ready signal was ever seen). Then mark the
worker Ready=true and verify the proxy resumes decode_forward.

Pure proxy decision unit test, TestClock for the time window, no SIP
traffic needed.

## Acceptance criteria

1. All three new fake tests pass under `npm run test:fake`.
2. `npm run test:k8s -- tests/k8s/proxy-drain.test.ts` passes 10/10
   consecutive runs.
3. New Prometheus counter is queryable: 
   `sipfp_decode_forward_promoted_total{from="not-ready"} > 0` after a 
   proxy-drain run.
4. No regression in existing K8s suites (`proxy-failover-*.test.ts`).

## Out-of-scope (explicitly)

  - **Slice C** (primary-return rehydration of bak: takeover writes into
    pri: on the original primary). That's a separate concern: it's about
    DATA correctness after the new primary becomes ready and the
    survivor stops handling its calls. The current failure mode happens
    BEFORE the primary becomes ready, so Slice C doesn't apply.
  - **Worker-side defenses against unknown in-dialog requests.** The
    worker correctly returns 481 today. The fix is upstream in the
    proxy, not downstream in the worker.
  - **Mocked sipp.** The fake stack already constructs SIP requests
    programmatically; we don't need to embed a sipp simulator.

## Why we caught this only at K8s scale

In existing fake-stack tests, when a worker is "respawned" (the test
rebuilds the layer), all peer state is re-seeded BEFORE the worker
resumes serving. The "boot window where the proxy can route to a
NotReady pod" doesn't exist in fake stacks because there's no
intermediate moment between layer-rebuild and the worker becoming Ready.
K8s exposes that window organically (StatefulSet recreates the pod
faster than ReadyGate completes, especially under cold-cluster
conditions where peer enumeration takes longer). The new Test 1 above
re-creates this window deterministically by holding the worker's
WorkerReadiness false while the proxy attempts to route to it.
