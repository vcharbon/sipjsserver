## Scope notes for RFC 3264

The B2BUA is **signaling-only**. The audit channel observes SIP
messages and their SDP bodies — it does not observe RTP / RTCP
packets, nor media flow direction at the wire level. Two policies
follow from this and are reflected in the classification below:

- **SDP-content + offer/answer model MUSTs**: in scope. Anything
  observable from SDP body bytes and the offer→answer (or
  offer→answer→re-offer) message ordering is auditable. Will-implement
  rules cover this surface.
- **RTP / RTCP / codec selection MUSTs**: out of scope. Verifying
  these would need media-plane probes (RTP receivers, payload-type
  extraction on RTP frames, RTCP listener state) that the harness
  does not have. Classifying these as out-of-scope is honest about
  what the audit channel can prove. The rule taxonomy carries
  `out-of-scope: media-plane` for these rows.

Additional B2BUA-specific notes:

- Multicast is not a deployment target. Multicast-specific MUSTs in
  §6.2 classify `out-of-scope: not-in-deployment`.
- Capability-indication SDPs (§9) are an OPTIONS-response feature
  not used by this B2BUA. Capability §9 MUSTs classify
  `out-of-scope: capability-indication-not-emitted`.
- The legacy RFC 2543 hold idiom (`c=0.0.0.0`) is rare but valid;
  §8.4 MUSTs that constrain it are in scope.

The `rfc.sdpOriginContinuity` cross-message rule (currently advisory
per RuleManifest.md) covers the §8 `o=` line invariants — see
RFC3264-MUST-039 / RFC3264-MUST-040 below.

## Conventions

- **MUST-ID**: `RFC3264-MUST-NNN`, numbered in document order.
- **Subject**: `uac` / `uas` / `proxy`. A B2BUA covers `{uac, uas}`
  per call.
- **Classification**: `will-implement` / `already-implemented` /
  `justified-not-implemented` / `restatement`.
- **Rule / justification**: planned rule name, existing rule name, or
  `<taxonomy-keyword>: <one-line reason>`.

## Phase 2 sequencing

Per the plan, Phase-2 PRs are scoped per RFC section. For RFC 3264:

1. **§4 (Protocol Operation)** — 1 will-implement MUST (M-001/M-002,
   one rule). Glare prevention.
2. **§5 (Generating the Initial Offer)** — 5 will-implement MUSTs
   (M-003 through M-006 + M-012). One rule: `rfc.sdpBodyParseable`.
   Plus §5's media-plane and intent-conditional rows are
   justified-not-implemented (no Phase-2 work).
3. **§6 (Generating the Answer)** — 5 will-implement MUSTs across 5
   distinct rules: `rfc.answerMLineCountMatchesOffer`,
   `rfc.answerTLineEqualsOffer`, `rfc.answerMediaTypeMatchesOffer`,
   `rfc.directionPairValid`, `rfc.rejectedStreamMinimalAnswer`. The
   largest Phase-2 PR for 3264.
4. **§8 (Modifying the Session)** — 4 will-implement MUSTs across 3
   rules: `rfc.reOfferMLineCountMonotonic` (M-042 + M-043),
   `rfc.zeroPortPropagation` (M-044),
   `rfc.payloadTypeMappingStable` (M-047). Also `M-039` /`M-040` are
   `already-implemented` via `rfc.sdpOriginContinuity` — Phase 2
   confirms the rule truly enforces them and lifts the advisory
   downgrade if fixtures permit.
5. **§8.4 (Hold)** — 1 will-implement MUST (M-051), narrow legacy
   case. Likely co-lands with §8.

§§5.1, 6.2, 7, 8.3.1-8.3.4, 9, 11 are entirely
`justified-not-implemented` — no Phase 2 work.

Helper extraction: every cross-message rule listed here is a
candidate first consumer of the planned `_offer-answer.ts` helper
(see [RFC_Verification.md](../RFC_Verification.md#helper-extraction-policy)).
The first §6 rule to land triggers extraction; subsequent rules in
the same PR reuse it. `rfc.sdpBodyParseable` is a peer rule and does
not need the helper.
