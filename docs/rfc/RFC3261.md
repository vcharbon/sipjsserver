## Scope notes for RFC 3261

RFC 3261 is the core SIP specification. ~580 raw grep hits compress
to 191 unique entries after restatement consolidation. The
classification reflects scope decisions locked with the user:

- **§7 (Messages, headers, bodies)**: most MUSTs are SDP-message
  grammar and header field syntax — `parser-enforced` by the custom
  SIP parser. The few semantic ones (Content-Length, Content-Type
  presence) are already covered by shipped peer rules.
- **§8 (UA Behavior)**: largest single coverage area. Most MUSTs are
  `already-implemented` by the 19 shipped peer rules + 7 shipped
  cross-message rules.
- **§9 (CANCEL)**: most MUSTs covered by `rfc.cancelRequestUri` +
  `rfc.cancelViaBranch`. A few new will-implement rules pick up
  CANCEL-specific Require/Proxy-Require + 1xx-precondition rules.
- **§10 (REGISTER)**: registrar-side MUSTs are
  `out-of-scope: not-a-registrar` (B2BUA doesn't run a registrar).
  UAC-side REGISTER pass-through MUSTs are in scope.
- **§11 (OPTIONS)**: partial coverage; in-scope MUSTs become
  will-implement.
- **§12 (Dialogs)**: dialog-formation MUSTs are mostly
  `arch-guaranteed` by the B2BUA dialog tracker. Route-set and
  in-dialog request rules are already-implemented by
  `rfc.midDialogRoute`, `rfc.midDialogFromUri`.
- **§13-15 (INVITE, modification, BYE)**: mix of already-implemented
  rules (offer/answer enforced by OfferAnswerTracker; tag/CSeq/Contact
  by existing peer rules) and a few new will-implement rules.
- **§16 (Proxy Behavior)**: scope decision locked: B2BUA workers are
  audited as `proxy` too (subject includes `proxy` for relevant
  MUSTs), not just dedicated proxy binds. Several MUSTs already-
  implemented (`rfc.recordRoute`, `rfc.maxForwards`,
  `rfc.proxy100TryingNotForwarded`), rest cluster into a small group
  of new rules.
- **§17 (Transactions)**: almost entirely `transport-enforced` (the
  transaction state machine is library code; its correctness is
  ADR-guaranteed by the transaction layer's tests, not by the audit
  channel).
- **§18 (Transport)**: TCP/UDP MUSTs are `arch-guaranteed`. The few
  observable ones (Via `received` parameter on cross-NAT receipt)
  are `already-implemented` indirectly via the Via rule.
- **§19-20 (URI schemes, header fields)**: parser-enforced. The
  custom SIP parser rejects malformed inputs at ingest (see ADR-0007).
- **§21 (Status codes)**: status code semantics — partial coverage
  by existing 4xx-response rules. A few new will-implement rules
  pick up 404/405/415/420 response shape.
- **§22 (Authentication)**: `out-of-scope: security-delegated` —
  trusted-backbone deployment. The B2BUA neither generates nor
  validates digest auth.
- **§23 (S/MIME)**: `out-of-scope: not-deployed` — no S/MIME support.
- **§26 (Security)**: `out-of-scope: security-delegated`.
- **§8.1.3.4 (3xx Processing)**: `out-of-scope: 3xx-not-followed` —
  B2BUA worker does not recursively follow 3xx redirects; surfaces
  upstream (locked with user 2026-05-25).

## Conventions

- **MUST-ID**: `RFC3261-MUST-NNN`, numbered in document order.
- **Subject**: `uac` / `uas` / `proxy`. B2BUA covers `{uac, uas}` per
  call; `{uac, uas, proxy}` for MUSTs the user locked as also-on-
  workers.
- **Classification**: `will-implement` / `already-implemented` /
  `justified-not-implemented` / `restatement`.

## Phase 2 sequencing

Per the plan, Phase-2 PRs are scoped per RFC section. For RFC 3261:

1. **§8.1 (UAC headers)** — most already-implemented; M-016
   (`rfc.noToTagOnInitialRequest`) shipped 2026-05-25.
2. **§8.2 + §21 (UAS error responses)** — group rule landings for
   M-030, M-033, M-036, M-037, M-182 (405/420/415/extension shape).
3. **§9 (CANCEL)** — M-045, M-046, M-048 (cseq method, Route echo,
   1xx-precondition).
4. **§10 (REGISTER)** — M-051, M-052, M-054 (no Route, header shape,
   serial REGISTER).
5. **§11 (OPTIONS)** — M-059.
6. **§12.2.2 (Unknown dialog)** — M-071 (`rfc.unknownDialog481`).
7. **§14 (Re-INVITE)** — M-083, M-086 (concurrent INVITE 500/491).
8. **§15 (BYE)** — M-089 (`rfc.noByeOutsideOrEarlyDialog`).
9. **§16 (Proxy)** — M-095, M-100, M-105, M-113 (proxy 100-Trying
   timing, strict-route rewriting on receive, no-target 404,
   strict-route shuffle on send).
10. **§17.1.1.3 (ACK construction)** — M-145
    (`rfc.ackPreservesInviteRoute`).

Most §17, §18, §19-20, §22-26 are already-implemented or
justified-not-implemented — no Phase-2 work there.

The 14 will-implement rules combine with the 16 planned for RFC 3262
and 11 planned for RFC 3264 to form the Phase-2 backlog. See
[RuleManifest.md](RuleManifest.md) for the unified roster.

## Note on "will-implement" count vs Phase 2 planning

Some inventory rows above carry the same planned rule name (e.g.
`rfc.unsupportedExtension420` covers M-033, and `rfc.unsupportedExtension421`
covers M-182 — closely related but distinct status codes). The
manifest groups these into individual rules where each fires on its
own status code; reviewer review of the manifest before Phase 2
finalizes consolidation.
