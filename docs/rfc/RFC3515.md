# RFC 3515 — The Session Initiation Protocol (SIP) Refer Method

Per-MUST inventory. Source-of-truth grep:
[_raw/RFC3515-musts.txt](_raw/RFC3515-musts.txt). RFC text:
[https://www.rfc-editor.org/rfc/rfc3515.txt](https://www.rfc-editor.org/rfc/rfc3515.txt).

See [docs/RFC_Verification.md](../RFC_Verification.md) for the
classification taxonomy and lifecycle. Planned rules appear in
[RuleManifest.md](RuleManifest.md).

## Conventions

- **MUST-ID**: `RFC3515-MUST-NNN`, numbered in document order. Stable —
  reference these from rule code comments.
- **Subject**: `uac` / `uas` / `proxy`. A B2BUA covers `{uac, uas}` per
  call (A-leg as UAS, B-leg as UAC). For REFER specifically, the B2BUA
  in this codebase is only ever the REFER **recipient** (UAS face) —
  A-leg REFER is rejected with 501, B-leg REFER is intercepted (see
  [src/b2bua/rules/defaults/TransferRules.ts](../../src/b2bua/rules/defaults/TransferRules.ts)).
  REFER-issuer (`uac` subject) obligations apply to the peer that
  sends REFER, audited via the peer bind.
- **Classification**: one of `will-implement`, `already-implemented`,
  `justified-not-implemented`, `restatement`. No bare `todo`.

## Scope decisions

- **REFER-issuer obligations (uac subject)** apply to peer binds. The
  B2BUA never issues REFER in this codebase; A-leg REFER is rejected
  501, B-leg REFER is intercepted. M-011 / M-016 carry
  `out-of-scope` for the DUT subject narrowing but still trip on a
  REFER-issuing peer that misbehaves.
- **Implicit-subscription mechanics** (M-007 "create a subscription
  and send notifications") are partly observable. The "create" half
  is agent-internal; the "send notifications" half is observable as
  *at least one NOTIFY following the 2xx REFER on the same dialog*,
  which is what the planned rule asserts.
- **Approval semantics** (M-004 "MUST contact the resource
  identified by the Refer-To URI") are agent-internal. Plus the
  B2BUA delegates the contact decision to the backend
  (`POST /call/refer`) — outbound INVITE-to-target is application
  policy, not the strict 3515 obligation. Classified
  `unobservable-post-hoc`.
- **Multiple-REFER `id` parameter** (M-015) is `out-of-scope` because
  the B2BUA explicitly rejects a second concurrent REFER with 491
  (see `transferRejectSecondReferRule` in
  [TransferRules.ts](../../src/b2bua/rules/defaults/TransferRules.ts)).
  The rule would never fire on the DUT face.
- **SDP / media-plane** is unaffected by REFER itself; nothing to add.

## Phase 2 sequencing

The 9 `will-implement` rules cluster into three sub-batches; suggest
landing in this order:

1. **Refer-To shape + 400 path** — M-003, M-005. Peer rule
   (`rfc.referToExactlyOne`) plus cross-message
   (`rfc.referMalformed400`). Pairs naturally.
2. **REFER acceptance + NOTIFY emission** — M-006, M-007, M-009,
   M-010, M-014. These are the bulk of the B2BUA's UAS-face REFER
   contract; the NOTIFY-shape peer rules can share a sipfrag-body
   parsing helper (candidate `_sipfrag.ts` — defer to second consumer
   per the helper-extraction policy).
3. **Subscription lifecycle and SUBSCRIBE rejection** — M-012, M-017.
   Both touch subscription-state semantics; small and self-contained.

`justified-not-implemented` taxonomy distribution:

- `unobservable-post-hoc`: 2 (M-004, M-011)
- `out-of-scope`: 2 (M-015, M-016)

No `arch-guaranteed` / `parser-enforced` justifications surfaced
beyond the cardinality embedded in M-001's note.

## Note on cross-reference to existing rules

- M-001 / M-002 fold into existing RFC 3261 rules (`rfc.contactPresence`
  / `rfc.midDialogRoute`). No new manifest rows for these.
- M-006's lift-out from `rfc.unknownDialog481`-shadowing was the
  motivating example for queuing RFC 3515 first in Phase 3.
