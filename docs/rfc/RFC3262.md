## Conventions

- **MUST-ID**: `RFC3262-MUST-NNN`, numbered in document order. Stable —
  reference these from rule code comments.
- **Subject**: `uac` / `uas` / `proxy` (or a comma-separated set). A
  B2BUA covers `{uac, uas}` per call (A-leg as UAS, B-leg as UAC).
- **Classification**: one of `will-implement`, `already-implemented`,
  `justified-not-implemented`, `restatement`. No bare `todo`.
- **Rule / justification**: planned rule name (for `will-implement`),
  existing rule name (for `already-implemented`), or
  `<taxonomy-keyword>: <one-line reason>` (for
  `justified-not-implemented`).

## Phase 2 sequencing

Per the plan, Phase-2 PRs are scoped per RFC section. For RFC 3262
that resolves to:

1. **§3 (UAS Behavior)** — 14 `will-implement` MUSTs across 11 planned
   rules. Largest landing.
2. **§4 (UAC Behavior)** — 5 `will-implement` MUSTs across 4 planned
   rules.
3. **§5 (Offer/Answer Model and PRACK)** — 3 `will-implement` MUSTs
   collapsed into 1 planned rule (`rfc.prackOfferAnswerModel`).
   Likely co-lands with the first RFC 3264 cross-message helper —
   shared offer/answer state-walking primitives → `_offer-answer.ts`
   per the [helper-extraction policy](../RFC_Verification.md#helper-extraction-policy).

Each PR also lands the positive-case fixture(s) the rules need to
demonstrate they fire (see process doc §"Every landed rule must
demonstrably fire").
